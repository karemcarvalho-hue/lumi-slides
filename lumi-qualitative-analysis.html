Parte 1: Problemas que Exigem Mudancas no Backend (Cirrus / Orquestrador Lumi)

B1. Dead-Ends Massivos -- Conversas morrem sem proximo passo

Impacto: 254 threads (12.2% dos bad/mixed) Prioridade: P0

Fonte: lumi-annotations LLM -- filtrar observations[].type == "dead-end"

O que acontece: - 409 conversas terminam sem resolucao e sem oferecer proximo passo - 92 casos Lumi nao consegue executar a tarefa e nao redireciona - 16 casos Lumi diz que vai derivar a soporte mas nao envia o link

Causa raiz: O orquestrador nao tem fallback robusto. Quando nenhum subAgent consegue resolver, Lumi congela ou da resposta generica.

O que fazer no backend: 1. Implementar fallback chain no orquestrador: se nenhum subAgent resolve -> buscar no helpjuice -> oferecer link de suporte -> nunca terminar sem proximo passo 2. Garantir que a derivacao a soporte sempre inclua o link funcional (endpoint de handoff precisa ser validado) 3. Adicionar mecanismo de deteccao de "conversa presa" (ex: 2+ turnos sem acao util = escalar automaticamente)

Evidencia humana:

"Dijo que derivaba la conversacion a support pero no paso el link" -- luis.martin, threads a7c91828 e 1f5a6743 (lumi-annotations-2026-03-06.json)

B2. Guardrails Excessivos -- Lumi recusa quando deveria agir

Impacto: 101 threads (4.9% dos bad/mixed) Prioridade: P1

Fonte: lumi-annotations LLM -- filtrar observations[].type == "unnecessary-guardrail"

O que acontece: - 149 casos: Lumi deveria perguntar mais info em vez de recusar - 13 casos: diz "nao tenho capacidade" mas de fato tem o subAgent disponivel - 9 casos: recusa acoes de catalogo que sao validas - 8 casos: recusa configuracoes que poderia explicar via helpjuice

Causa raiz: Prompt do orquestrador e conservador demais. Prefere rejeitar a pedir clarificacao.

O que fazer no backend: 1. Revisar o system prompt do orquestrador: trocar comportamento default de "rejeitar" para "pedir clarificacao" 2. Antes de dizer "nao posso", verificar se existe artigo no helpjuice sobre o tema 3. Implementar logica de "tool availability check" -- nao dizer "nao tenho capacidade" quando o subAgent esta disponivel para a loja

B4. Falta de Awareness do Contexto da Loja

Impacto: 108 ocorrencias de missing-context + impacto indireto em stats e features Prioridade: P2

Fonte: lumi-annotations LLM -- filtrar observations[].type == "missing-context"

O que acontece: - Lumi tenta usar Stats Agent em lojas que nao tem stats habilitado - Nao sabe quais features/planos a loja tem ativo - Nao tem contexto do storefront, promocoes ativas, meios de pagamento e envio da loja

O que fazer no backend: 1. Injetar no contexto do orquestrador: plano da loja, features habilitadas, meios de pagamento/envio configurados 2. Antes de acionar stats/orders agent, validar se a loja tem acesso 3. Considerar endpoint que retorna "capabilities" da loja para enriquecer o prompt

Evidencia humana:

"El merchant no tiene disponible stats" -- luis.martin, threads 0a3e681a, 8f50ad68, e487ed65 (lumi-annotations-2026-03-06.json)

"No tiene el contexto completo de la tienda para poder responder" -- luis.martin, thread cfe7af8c (lumi-annotations-2026-03-06.json)

B5. Hallucinations em Geracao de Conteudo

Impacto: 20 ocorrencias (0.5% -- baixo mas grave) Prioridade: P3

Fonte: lumi-annotations LLM -- filtrar observations[].type == "hallucination"

Onde ocorre: - Geracao de SEO (titulo/descricao) - Descricoes de produto - Consultas de preco/dados de catalogo

O que fazer no backend: 1. Na geracao de SEO/descricoes: validar que os dados do produto existem antes de gerar 2. Em consultas de preco: nunca inventar valores -- retornar apenas dados confirmados pelo catalogo 3. Adicionar disclaimer automatico quando geracao e baseada em dados parciais

Parte 2: Problemas que Exigem Mudancas no Frontend / UX (Stratus)

F1. Respostas Desalinhadas -- Lumi explica quando deveria executar

Impacto: 365 threads (17.5% dos bad/mixed) -- maior problema isolado Prioridade: P0

Fonte: lumi-annotations LLM -- filtrar observations[].type == "misaligned-response"; cruzar com campo "featureSignals[].title" para a tabela abaixo

O que acontece: O merchant pede "crie uma promocao" e Lumi responde com um passo-a-passo de como criar. O merchant quer acao, nao explicacao.

Cruzamento com feature signals:

O que merchant pede Ocorrencias Como Lumi falha
Criar promocoes 58 Explica como fazer em vez de criar
Editar storefront 38 Responde com instrucoes genericas
Editar design da loja 23 Misaligned-response (19), wrong-routing (4)
Configurar la tienda 21 Da passo-a-passo que o merchant nao consegue seguir
Stats/metricas 24 Manda navegar em vez de mostrar dados

O que fazer (Backend + UX juntos):

Backend: 1. Refinar prompt para distinguir intents: "quero que FACA" vs "quero ENTENDER como fazer" 2. Quando Lumi nao pode executar mas pode explicar, ser explicito: "Aun no puedo crear promociones directamente, pero te explico como hacerlo en 3 pasos:" 3. Implementar confirmacao antes de agir: "Queres que cree la promocion con estos datos? [Si/No]"

Frontend/UX: 1. Quando Lumi envia instrucoes passo-a-passo, renderizar como checklist interativo (nao wall of text) 2. Adicionar botoes de acao contextual nas respostas (ex: "Ir a Promociones", "Crear producto") 3. Considerar action cards embarcados na resposta que linkem direto a tela relevante no admin

F2. Verbosidade -- Respostas longas demais

Impacto: 225 threads (10.8% dos bad/mixed) Prioridade: P1

Fonte: lumi-annotations LLM -- filtrar observations[].type == "verbose-response"; cruzar com campo "subAgents" para o breakdown abaixo

Contexto onde e pior: - SubAgent support: 192 ocorrencias - SubAgent catalog: 134 ocorrencias - Sem agent (resposta generica): 119 ocorrencias

O que fazer:

Backend (prompt): 1. Instrucao explicita no prompt: respostas de no maximo 3 paragrafos 2. Se precisar de mais detalhe, oferecer: "Queres que te explique mas sobre alguno de estos puntos?" 3. Proibir padding linguistico (ex: "Entendo tu consulta, voy a ayudarte con eso, permiteme explicarte...")

Frontend/UX: 1. Se resposta excede X caracteres, truncar com "Ver mas" expandivel 2. Para instrucoes longas, usar formato de accordion/stepper em vez de texto corrido 3. Considerar indicador visual de extensao da resposta

Recomendacao para o time

1. Adicionar ao ciclo atual (quick wins de 1-3 dias):

Estas acoes sao de esforco baixo e impacto altissimo. Nao competem com as features em andamento -- sao ajustes no orquestrador/prompt:

  • Fallback robusto: Lumi nunca termina uma conversa sem oferecer proximo passo (link de suporte, busca no helpjuice, ou pergunta de clarificacao). Impacta ~254 threads.
  • Prompt anti-verbosidade: Limitar respostas a 3 paragrafos max. Eliminar padding linguistico. Se precisar mais, perguntar. Impacta ~225 threads.
  • Prompt anti-guardrail: Trocar o default de "nao posso" por "me conta mais sobre o que precisa". Impacta ~101 threads.
  • Auditar Navigator: Revisar o mapeamento de URLs -- 61.8% das navegacoes falham. Impacta ~70 threads.

2. Encaixar no proximo ciclo (complementam as features em andamento):

Estas acoes melhoram a experiencia das features novas que estao sendo construidas:

  • Distinguir "faz pra mim" vs "me explica" no prompt do orquestrador. Sem isso, as features novas de catalogo vao sofrer do mesmo problema: merchant pede, Lumi explica. Impacta ~365 threads.
  • Awareness de features: Antes de acionar stats/catalog, validar que a loja tem acesso. Evita frustracoes desnecessarias. Impacta ~108 threads.
  • Truncamento de respostas longas no front ("Ver mas"). Impacta ~225 threads.
  • Error state claro quando Lumi nao pode resolver. Impacta ~254 threads.

3. Reavaliar priorizacao do Workbook:

Manter o que ja esta In Progress/Review (AIL-7, AIL-1, AIL-100, AIL-101, AIL-102). Considerar nao iniciar as issues ainda em Todo (AIL-17 filtros, AIL-88 highlights) ate que os problemas P0/P1 de qualidade base estejam encaminhados.

Por que isso importa

Sem resolver os problemas de qualidade base, as features novas vao herdar os mesmos padroes de falha. Um Lumi que sabe criar produtos mas continua dando respostas verbose, morrendo em dead-ends e recusando com guardrails excessivos nao vai ter a adocao esperada.

A ordem importa: primeiro garantir que o que ja existe funciona bem, depois expandir capacidades.
